# Add OAuth2 Authentication (GCP) -- Change Specification

> Generated on: 2026-03-22
> Project: claude-code-webui
> Version: 1.1
> Status: Draft
> Type: Change Specification

## 1. Change Summary

Add Google OAuth2 authentication to the FastAPI backend so that users must authenticate via their Google account before accessing any API endpoint (except health checks). The server currently has no authentication; all endpoints are publicly accessible. User profiles obtained from Google (email, name, picture) are persisted in a SQLite `users` table via `aiosqlite`. The authentication middleware must protect the WebSocket chat endpoint and file upload endpoint (defined in the web-chat-and-file-upload spec) but must NOT protect the MCP stdio subprocess channel (which is a local process, not an HTTP endpoint). The frontend uses htmx for login/logout UI flows.

## 2. Current State Analysis

### 2.1 Project Overview
Claude Code Web UI is a Python 3.13 FastAPI application providing a web interface for Claude Code CLI. It uses Typer for CLI, pydantic-settings for configuration (env prefix `CCWEBUI_`), OpenTelemetry for tracing, and Rich for console logging. The project is at an early stage: only a `/health` endpoint exists. There is no authentication, no session management, and no user model.

### 2.2 Existing Specifications
- `specs/BACKLOG.md`: Lists planned features including OAuth2 Authentication (GCP), Web Chat Interface, MCP Server Integration, Claude Code Channel Protocol, and File Upload.
- `specs/2026-03-22_20:34:04-web-chat-and-file-upload.md`: Defines the WebSocket chat endpoint (`/ws/chat`), file upload endpoint (`POST /api/files/upload`), chat frontend (htmx), Claude CLI integration, and in-memory conversation history. The frontend uses htmx for all dynamic interactions.
- `specs/2026-03-22_20:35:31-mcp-channel-integration.md`: Defines MCP server integration via subprocess (stdio transport), not an HTTP endpoint.

### 2.3 Relevant Architecture
- **API layer** (`src/api.py`): `create_app()` factory builds the FastAPI app, adds a `/health` endpoint, instruments with OTel. The module-level `app = create_app()` is used by uvicorn.
- **Config** (`src/config.py`): `Settings` class with `CCWEBUI_` prefix. Currently has `app_name`, `debug`, `host`, `port`.
- **CLI** (`src/claude_code_webui.py`): Typer app with `serve` command that runs uvicorn.
- **No dependencies** on authlib, google-auth, or any auth library exist today.
- **No middleware** is registered on the FastAPI app.

## 3. Requested Modifications

### MOD-001: OAuth2 Configuration Settings
- **Type:** Add
- **Description:** Add GCP OAuth2 settings to the `Settings` class: client ID, client secret, redirect URI, allowed email domains (optional allowlist), and session secret key.
- **Rationale:** OAuth2 credentials and session configuration must be externalized via environment variables, following the existing `CCWEBUI_` prefix pattern.
- **Priority:** Must-have
- **Details:**
  - `oauth2_client_id: str` (required, no default)
  - `oauth2_client_secret: str` (required, no default)
  - `oauth2_redirect_uri: str` (required, no default; e.g. `http://localhost:8080/auth/callback`)
  - `oauth2_allowed_domains: list[str]` (optional; empty list means all domains allowed)
  - `session_secret_key: str` (required, no default; used for signing session cookies)
  - The application must fail fast at startup if any required OAuth2 field is missing.

### MOD-002: OAuth2 Login/Callback Endpoints
- **Type:** Add
- **Description:** Add `GET /auth/login` and `GET /auth/callback` endpoints implementing the OAuth2 Authorization Code flow with Google.
- **Rationale:** Standard browser-based OAuth2 flow: redirect to Google consent screen, receive authorization code, exchange for tokens, extract user identity.
- **Priority:** Must-have
- **Details:**
  - `GET /auth/login`: Generate a state token, store it in the session, redirect user to Google's authorization endpoint with scopes `openid email profile`.
  - `GET /auth/callback`: Validate the state token, exchange the authorization code for an access token via Google's token endpoint, fetch user info from Google's userinfo endpoint, validate the email domain against `oauth2_allowed_domains` (if configured), create a session with user data, redirect to the root `/`.
  - On failure (invalid state, token exchange error, domain not allowed): return a 403 response with a clear error message.

### MOD-003: Session Management
- **Type:** Add
- **Description:** Add cookie-based session management using `starlette.middleware.sessions.SessionMiddleware` with a signed cookie. User profiles are persisted in SQLite via `aiosqlite` (see MOD-007).
- **Rationale:** Sessions are needed to persist authenticated user state across requests. Starlette's built-in session middleware handles the session cookie. SQLite provides durable user profile storage so user data survives server restarts and can be queried by other features (e.g., conversation ownership).
- **Priority:** Must-have
- **Details:**
  - Use `SessionMiddleware` with `session_secret_key` from settings.
  - Session cookie stores: `user_email`, `authenticated` (bool). Minimal data in the cookie; full profile is fetched from SQLite on demand.
  - Session cookie: `httponly=True`, `samesite=lax`.
  - On successful OAuth2 callback, upsert the user profile (email, name, picture) into the SQLite `users` table (see MOD-007), then set the session cookie.

### MOD-004: Authentication Middleware
- **Type:** Add
- **Description:** Add a FastAPI dependency or middleware that checks whether the user is authenticated before accessing protected endpoints. The middleware must also protect cross-spec endpoints: the WebSocket chat (`/ws/chat`) and file upload (`POST /api/files/upload`) from the web-chat-and-file-upload spec. It must NOT protect the MCP stdio channel (subprocess, not HTTP).
- **Rationale:** All HTTP endpoints except `/health`, `/auth/login`, and `/auth/callback` must require authentication. The MCP channel is a local subprocess communicating via stdin/stdout, so it is inherently not exposed over HTTP and does not need HTTP-level auth.
- **Priority:** Must-have
- **Details:**
  - Implement as a FastAPI dependency (`get_current_user`) that reads the session and returns user data or raises `HTTPException(401)`.
  - Apply the dependency to the FastAPI app via `app.dependency_overrides` or a router-level dependency.
  - Public endpoints (`/health`, `/auth/login`, `/auth/callback`, `/auth/logout`) must be excluded from authentication checks.
  - **Protected cross-spec endpoints** (from web-chat-and-file-upload spec):
    - `/ws/chat` (WebSocket): The WebSocket handshake must verify authentication. If the session is not authenticated, reject the WebSocket connection with close code 4001.
    - `POST /api/files/upload`: Standard HTTP auth check via `get_current_user` dependency.
  - **Excluded from auth** (from mcp-channel-integration spec):
    - The MCP stdio channel is a subprocess spawned by the server. It communicates via stdin/stdout pipes, not HTTP. No auth middleware applies.
  - Unauthenticated API requests must receive a 401 JSON response: `{"detail": "Not authenticated"}`.
  - Unauthenticated browser requests (Accept: text/html) must redirect to `/auth/login`. For htmx requests (identified by `HX-Request: true` header), return a 401 response with `HX-Redirect: /auth/login` header so htmx handles the redirect client-side.

### MOD-005: Logout Endpoint
- **Type:** Add
- **Description:** Add `GET /auth/logout` to clear the session and redirect to `/auth/login`. The logout button in the frontend uses htmx (`hx-get="/auth/logout"`).
- **Rationale:** Users must be able to sign out.
- **Priority:** Must-have
- **Details:**
  - Clear all session data.
  - For standard requests: redirect (302) to `/auth/login`.
  - For htmx requests (`HX-Request: true`): return 200 with `HX-Redirect: /auth/login` header.
  - This endpoint is public (no auth required).

### MOD-006: User Info Endpoint
- **Type:** Add
- **Description:** Add `GET /auth/me` to return the current user's profile data. Data is fetched from the SQLite `users` table using the email from the session cookie.
- **Rationale:** The frontend (htmx) will need to display user information (name, picture) and check auth status.
- **Priority:** Should-have
- **Details:**
  - Returns JSON: `{"email": "...", "name": "...", "picture": "..."}` (data from SQLite `users` table).
  - Requires authentication (returns 401 if not authenticated).
  - The htmx frontend can call this endpoint to render user profile fragments.

### MOD-007: SQLite User Persistence
- **Type:** Add
- **Description:** Add a SQLite database (via `aiosqlite`) with a `users` table to persist user profiles obtained from Google OAuth2. This is the project's first use of SQLite and establishes the persistence pattern for other features.
- **Rationale:** Persisting user data in SQLite ensures profiles survive server restarts, enables future features like conversation ownership, and avoids storing large profile data in cookies.
- **Priority:** Must-have
- **Details:**
  - New setting: `database_url: str` (default: `sqlite:///data/ccwebui.db`), configurable via `CCWEBUI_DATABASE_URL`.
  - New module: `src/database.py` providing async database lifecycle (init, close) and query helpers.
  - **Schema** (`users` table):
    - `email: TEXT PRIMARY KEY` (Google email, unique identifier)
    - `name: TEXT NOT NULL` (display name from Google)
    - `picture: TEXT` (profile picture URL from Google, nullable)
    - `created_at: TEXT NOT NULL` (ISO 8601 timestamp of first login)
    - `last_login_at: TEXT NOT NULL` (ISO 8601 timestamp of most recent login)
  - **Upsert on login**: On successful OAuth2 callback, insert the user if new or update `name`, `picture`, and `last_login_at` if existing.
  - **Database initialization**: Create the `users` table on application startup (via the FastAPI lifespan context manager). Use `CREATE TABLE IF NOT EXISTS`.
  - **Connection management**: Use `aiosqlite` for async access. Open a connection pool (or single connection with WAL mode) during lifespan startup, close on shutdown.
  - The data directory (`data/`) must be created if it does not exist.

### 4.5 Dependencies & Risks
- **New dependency:** `authlib>=1.3.0` for OAuth2 client (handles authorization URL generation, token exchange). Mature, well-maintained library.
- **New dependency:** `itsdangerous>=2.2.0` for session cookie signing (required by Starlette's SessionMiddleware). Already a transitive dependency of Starlette but must be declared explicitly.
- **New dependency:** `aiosqlite>=0.20.0` for async SQLite access. Lightweight wrapper around Python's built-in `sqlite3` module.
- **New dependency:** `httpx` is already present; used by authlib's async HTTP client for token exchange.
- **Risk: GCP Console Setup:** Users must create OAuth2 credentials in GCP Console before the app can start. The spec must document this prerequisite clearly.
- **Risk: Startup failure:** The app will refuse to start if OAuth2 settings are missing. This is intentional (fail-fast) but will break existing deployments that do not set these variables. Mitigation: add an `oauth2_enabled: bool = False` toggle so the app can still run without OAuth2 for local development.
- **Risk: SQLite file permissions:** The database file directory (`data/`) must be writable by the application process. Docker deployments must mount a volume for persistence.
- **Cross-spec dependency:** The WebSocket chat endpoint and file upload endpoint (from the web-chat-and-file-upload spec) must integrate with the `get_current_user` dependency. Implementation of those features must account for auth when `oauth2_enabled=True`.
- **No impact on MCP:** The MCP stdio channel (from the mcp-channel-integration spec) is a subprocess and does not pass through HTTP middleware. No auth integration is needed or desired.
- **No breaking changes for existing consumers:** The `/health` endpoint remains public. No external APIs exist yet.
- **Rollback:** Remove `SessionMiddleware`, auth router, database module, and OAuth2 settings. Drop the `users` table or delete the SQLite file. Straightforward since auth is additive.

## 5. New & Modified Requirements

### New Requirements

#### FR-NEW-001: OAuth2 Configuration
- **Description:** The system must load OAuth2 settings from environment variables with prefix `CCWEBUI_`.
- **Inputs:** Environment variables: `CCWEBUI_OAUTH2_CLIENT_ID`, `CCWEBUI_OAUTH2_CLIENT_SECRET`, `CCWEBUI_OAUTH2_REDIRECT_URI`, `CCWEBUI_SESSION_SECRET_KEY`, `CCWEBUI_OAUTH2_ALLOWED_DOMAINS` (comma-separated), `CCWEBUI_OAUTH2_ENABLED`, `CCWEBUI_DATABASE_URL`.
- **Outputs:** Populated `Settings` object with validated fields.
- **Business Rules:**
  - When `oauth2_enabled` is `True`, `oauth2_client_id`, `oauth2_client_secret`, `oauth2_redirect_uri`, and `session_secret_key` must all be set; the app must raise a validation error at startup if any is missing.
  - When `oauth2_enabled` is `False` (default), OAuth2 fields are ignored and no auth middleware is applied.
  - `oauth2_allowed_domains` defaults to an empty list (all domains allowed).
- **Priority:** Must-have

#### FR-NEW-002: OAuth2 Login Redirect
- **Description:** The system must expose `GET /auth/login` that redirects users to Google's OAuth2 consent screen.
- **Inputs:** None (browser GET request).
- **Outputs:** HTTP 302 redirect to `https://accounts.google.com/o/oauth2/v2/auth` with query parameters: `client_id`, `redirect_uri`, `response_type=code`, `scope=openid email profile`, `state=<random_token>`.
- **Business Rules:**
  - A cryptographically random `state` token must be generated and stored in the session.
  - This endpoint is public (no authentication required).
  - This endpoint is only registered when `oauth2_enabled` is `True`.
- **Priority:** Must-have

#### FR-NEW-003: OAuth2 Callback
- **Description:** The system must expose `GET /auth/callback` to handle Google's OAuth2 redirect with the authorization code.
- **Inputs:** Query parameters: `code`, `state`.
- **Outputs:** On success: HTTP 302 redirect to `/`. On failure: HTTP 403 JSON response.
- **Business Rules:**
  - The `state` parameter must match the session-stored state; return 403 if mismatch.
  - Exchange the `code` for an access token via `https://oauth2.googleapis.com/token`.
  - Fetch user info from `https://www.googleapis.com/oauth2/v2/userinfo`.
  - If `oauth2_allowed_domains` is non-empty, the user's email domain must be in the list; return 403 otherwise.
  - Upsert the user profile (email, name, picture) into the SQLite `users` table (see FR-NEW-008).
  - Store `user_email` and `authenticated=True` in the session cookie. Full profile data is in SQLite.
- **Priority:** Must-have

#### FR-NEW-004: Session Management
- **Description:** The system must manage user sessions via signed HTTP cookies using Starlette's SessionMiddleware. The cookie stores minimal data (`user_email`, `authenticated`); full user profile is in the SQLite `users` table.
- **Inputs:** `session_secret_key` from settings.
- **Outputs:** Signed session cookie on every response.
- **Business Rules:**
  - Session cookie attributes: `httponly=True`, `samesite=lax`.
  - Session cookie data: `user_email`, `authenticated` (bool). Minimal cookie size; profile details fetched from SQLite.
  - The session middleware must be added before the auth router.
- **Priority:** Must-have

#### FR-NEW-005: Authentication Enforcement
- **Description:** The system must require authentication for all HTTP endpoints except: `/health`, `/auth/login`, `/auth/callback`, `/auth/logout`. This includes cross-spec endpoints `/ws/chat` and `POST /api/files/upload`. The MCP stdio channel (subprocess) is not an HTTP endpoint and is excluded.
- **Inputs:** Session cookie on incoming request.
- **Outputs:** Authenticated user data, or 401 response, or redirect to `/auth/login`.
- **Business Rules:**
  - Implement as a FastAPI dependency `get_current_user(request: Request) -> dict`.
  - If `session["authenticated"]` is `True`, fetch user profile from SQLite `users` table and return user data dict.
  - If not authenticated and the request is an htmx request (`HX-Request: true` header), return 401 with `HX-Redirect: /auth/login` header.
  - If not authenticated and `Accept: text/html` header is present (non-htmx), redirect (302) to `/auth/login`.
  - If not authenticated and request expects JSON, return `{"detail": "Not authenticated"}` with status 401.
  - For WebSocket (`/ws/chat`): verify authentication during the handshake. Reject with WebSocket close code 4001 if not authenticated.
  - When `oauth2_enabled` is `False`, no authentication is enforced (all endpoints are public).
- **Priority:** Must-have

#### FR-NEW-006: Logout
- **Description:** The system must expose `GET /auth/logout` that clears the session and redirects to `/auth/login`.
- **Inputs:** None.
- **Outputs:** HTTP 302 redirect to `/auth/login`.
- **Business Rules:**
  - Clear all session data.
  - This endpoint is public.
- **Priority:** Must-have

#### FR-NEW-007: User Info Endpoint
- **Description:** The system must expose `GET /auth/me` that returns the authenticated user's profile from the SQLite `users` table.
- **Inputs:** Session cookie (authenticated, contains `user_email`).
- **Outputs:** JSON: `{"email": "...", "name": "...", "picture": "..."}` (fetched from SQLite).
- **Business Rules:**
  - Requires authentication (use `get_current_user` dependency).
  - Returns 401 if not authenticated.
  - Reads user profile from SQLite `users` table by email.
- **Priority:** Should-have

#### FR-NEW-008: SQLite User Persistence
- **Description:** The system must persist user profiles in a SQLite database via `aiosqlite`.
- **Inputs:** User profile data from Google OAuth2 (email, name, picture).
- **Outputs:** Stored row in the `users` table.
- **Business Rules:**
  - The `users` table must have columns: `email` (TEXT, PRIMARY KEY), `name` (TEXT, NOT NULL), `picture` (TEXT, nullable), `created_at` (TEXT, ISO 8601, NOT NULL), `last_login_at` (TEXT, ISO 8601, NOT NULL).
  - On OAuth2 callback success, upsert the user: insert if new, update `name`, `picture`, and `last_login_at` if existing.
  - The database file location is configurable via `CCWEBUI_DATABASE_URL` (default: `sqlite:///data/ccwebui.db`).
  - The `users` table must be created on application startup using `CREATE TABLE IF NOT EXISTS`.
  - The database connection must be opened during FastAPI lifespan startup and closed on shutdown.
  - WAL mode must be enabled for concurrent read performance.
- **Priority:** Must-have

## 6. Non-Functional Requirements Changes

#### NFR-NEW-001: Security -- Session Cookie
- Session cookies must be signed with a cryptographically strong secret key (minimum 32 characters recommended in documentation).
- Cookies must be `httponly` and `samesite=lax`.

#### NFR-NEW-002: Security -- State Parameter
- The OAuth2 state parameter must use `secrets.token_urlsafe(32)` or equivalent to prevent CSRF attacks.

#### NFR-NEW-003: Security -- Secrets Not Logged
- OAuth2 client secret, session secret key, and access tokens must never appear in application logs or OTel traces.

#### NFR-NEW-004: Startup Validation
- When `oauth2_enabled` is `True`, the application must fail at startup with a clear error message if required OAuth2 settings are missing, rather than failing at first request.

#### NFR-NEW-005: SQLite Performance
- The SQLite database must use WAL (Write-Ahead Logging) mode for concurrent read access.
- Database queries must use parameterized statements (no string interpolation) to prevent SQL injection.
- The database connection must be managed via the FastAPI lifespan context manager (async open on startup, close on shutdown).

#### NFR-NEW-006: htmx Redirect Pattern
- When an htmx request (`HX-Request: true` header) hits an unauthenticated endpoint, the server must return a 401 status with an `HX-Redirect: /auth/login` header instead of a 302 redirect. This allows htmx to handle the full-page redirect correctly on the client side.

## 9. Consistency Notes

- The backlog in `specs/BACKLOG.md` lists "OAuth2 Authentication (GCP)" as a planned feature. This spec implements that item. The backlog entry should be marked as complete (checked) after implementation.
- **Cross-spec: web-chat-and-file-upload** (`specs/2026-03-22_20:34:04-web-chat-and-file-upload.md`): The WebSocket endpoint (`/ws/chat`) and file upload endpoint (`POST /api/files/upload`) defined in that spec must be protected by the OAuth2 middleware when `oauth2_enabled=True`. The chat frontend uses htmx, so auth redirects must use the `HX-Redirect` header pattern. The web-chat-and-file-upload spec does not mention authentication (it was written before this spec); implementations of both specs must coordinate on auth integration.
- **Cross-spec: mcp-channel-integration** (`specs/2026-03-22_20:35:31-mcp-channel-integration.md`): The MCP stdio channel is a subprocess communicating via stdin/stdout pipes. It is NOT an HTTP endpoint and must NOT have auth middleware applied. No changes to the MCP spec are needed.
- **SQLite as project-wide persistence**: This spec introduces SQLite (via `aiosqlite`) as the persistence layer. The web-chat-and-file-upload spec uses in-memory conversation history. Future specs may migrate conversation storage to SQLite, but that is out of scope for this change.

### GCP Setup Prerequisite
Users must:
1. Go to Google Cloud Console > APIs & Services > Credentials.
2. Create an OAuth 2.0 Client ID (Web application type).
3. Set the authorized redirect URI to match `CCWEBUI_OAUTH2_REDIRECT_URI` (e.g., `http://localhost:8080/auth/callback`).
4. Copy the client ID and secret to environment variables.

### Testing Strategy
All tests that call Google endpoints must mock HTTP calls using `httpx` mock or `respx`. No real Google API calls in tests. The `authlib` OAuth2 client uses `httpx` under the hood, so mocking at the `httpx` transport level is appropriate. Database tests must use an in-memory SQLite database (`:memory:`) for isolation and speed. Each test must get a fresh database instance.

## 11. Open Questions & TBDs

1. **Token refresh**: Should the server handle Google access token refresh for long-lived sessions? For now, the session persists until the cookie expires or the user logs out. Token refresh can be deferred.
2. **Session expiry**: What is the desired session lifetime? Starlette's SessionMiddleware uses browser-session cookies by default (expire when browser closes). A `max_age` setting could be added later.
3. **HTTPS enforcement**: In production, session cookies should be `secure=True` (HTTPS only). This could be tied to a `CCWEBUI_SECURE_COOKIES` setting. Deferred for now.
